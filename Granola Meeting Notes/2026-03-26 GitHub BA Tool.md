## What It Does
Log4brains is a docs-as-code knowledge base for logging Architecture Decision Records (ADRs) directly from your IDE. It provides a CLI to create ADRs from templates, a local preview server with hot reload, and a static site generator that publishes your decision log as a searchable website via GitHub Pages, GitLab Pages, or S3.

## Key Features
- Interactive CLI for creating new ADRs from customizable markdown templates (defaults to MADR format)
- Local dev server with hot reload for real-time preview while writing
- Static site generation with timeline view and full-text search
- Automatic metadata extraction from file content and git history
- No enforced numbering schema — prevents merge conflicts in team environments
- Multi-package/monorepo support for distributed projects
- CI/CD integration with GitHub Actions, GitLab CI, and Docker

## Installation
```bash
npm install -g log4brains
log4brains init        # Initialize in your project root
log4brains adr new     # Create a new ADR interactively
log4brains preview     # Start local preview server
log4brains build       # Generate static site for publishing
```

## How to Use It in Your BA Workflow (Obsidian)

### 1. Decision Log as a Living Knowledge Base
Run `log4brains init` in your project repo and use `log4brains adr new` each time a key decision is made in a meeting or workshop. Each ADR captures the context, decision, and consequences in structured markdown. In Obsidian, create a `Decisions/` folder that mirrors or links to your ADR directory — use `[[ADR-001 Choose Auth Provider]]` style links so your requirements notes, BRDs, and meeting notes can cross-reference the exact decision that drove a requirement.

### 2. Traceability from Requirements to Decisions
When writing a BRD or FRD in Obsidian, embed decision references inline: e.g., `> Per [[ADR-012]], we chose event-driven architecture for order processing`. Log4brains' generated site gives stakeholders a browsable, searchable version of the same decisions. This creates a two-tier system: Obsidian for your working BA notes with rich linking, and the Log4brains site as the published, stakeholder-facing decision record.

### 3. Stakeholder Communication — Published Decision Site
Run `log4brains build` and deploy to GitHub Pages. Share the URL with stakeholders who need to understand *why* certain technical or business decisions were made. The timeline view lets leadership see the cadence of decisions, while full-text search helps developers find the rationale behind constraints. As a BA, you become the curator of organizational knowledge — not just the documenter.

### 4. Gap Analysis via Decision Audit
Periodically review your ADR log in Obsidian by creating a `Decision Audit.md` note. List all ADRs and tag each as `#status/active`, `#status/superseded`, or `#status/needs-review`. Log4brains supports superseding decisions natively — when a new ADR replaces an old one, the link is preserved. This audit surfaces gaps: decisions that were deferred, areas with no documented rationale, or outdated choices that need revisiting. Example Obsidian query using Dataview: `TABLE status FROM "Decisions" WHERE type = "adr" SORT date DESC`.

### 5. Meeting-to-Decision Pipeline
After each requirements workshop or architecture review (captured via Granola or manual notes in Obsidian), extract any decisions made and immediately run `log4brains adr new` to formalize them. Use a consistent naming convention like `YYYYMMDD-decision-topic.md`. In your Obsidian meeting note, add a "Decisions Made" section with links: `- [[20260326-chose-api-gateway]]`. This ensures no decision gets lost between meeting notes and the formal record — and gives you an auditable trail from conversation to documentation.
