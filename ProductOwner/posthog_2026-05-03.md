# posthog

## Verdict
PostHog is worth a product owner's time if the team needs better evidence for roadmap, rollout, and discovery decisions. It is not a lightweight template repo: it is a full product analytics platform with session replay, surveys, feature flags, experiments, web analytics, data warehouse, and workflows. The blunt caveat is setup and instrumentation: without engineering support and clean event design, this becomes another dashboard nobody trusts.

## Why This Repo Was Picked
It won because it directly supports product-owner decisions after a feature ships, before a feature ships, and while a feature is being rolled out. A PO can use it to connect backlog bets to real usage, inspect failed journeys with session replay, ask users targeted questions through surveys, gate releases with feature flags, and validate impact through experiments. GrowthBook is stronger if the only job is experimentation; PostHog is stronger for a broader PO operating loop because analytics, feedback, rollout, and testing live in one system.

## What It Is
PostHog is an open-core product platform, not a framework document or curated list. The repository contains the main application for product analytics, web analytics, session replay, feature flags, experiments, surveys, data warehouse integrations, data pipelines, error tracking, LLM analytics, and automation workflows. It can be used through PostHog Cloud or self-hosted, with the repo README explicitly steering most users toward cloud and treating self-hosting as the advanced path.

## Why It Is Useful For Product Owners
- Backlog refinement: use funnel, retention, and path data to separate real user pain from internal noise.
- Prioritization: rank roadmap bets with usage, conversion, drop-off, survey, and experiment evidence.
- Discovery: combine surveys with session replay to understand why users fail or abandon workflows.
- Acceptance criteria: define measurable behavioral outcomes, not just "button exists" delivery checks.
- Experimentation: test changes against goal metrics before declaring a feature successful.
- Stakeholder alignment: show actual customer behavior and rollout results when priorities are disputed.
- Delivery coordination: use feature flags and cohorts to stage releases, monitor impact, and roll back bad changes faster.

## How I Would Actually Use It
1. Define one activation funnel for a key workflow, then use drop-off points to decide which backlog item is actually worth refinement.
2. Watch session replays for users who failed that funnel and turn repeated friction into specific story candidates.
3. Add a feature flag to a risky release and roll it out to internal users, beta customers, then a broader cohort.
4. Run an experiment for a pricing, onboarding, search, or conversion change and tie the result to a pre-agreed goal metric.
5. Trigger a short survey after a workflow failure or cancellation event instead of sending generic quarterly surveys.
6. Build a dashboard for the next roadmap review that shows adoption, retention, conversion, and release-impact metrics for recently shipped work.
7. Use cohorts to compare new users, power users, churn-risk users, and enterprise accounts before treating all feedback as equal.

## Limitations / Watch Outs
- This is too much tool if the team only needs a simple feedback form or roadmap board.
- Event instrumentation quality is everything. Bad event names and missing properties will make the analysis weak.
- Self-hosting is not a PO side project; it needs engineering ownership, infrastructure, upgrades, and data governance.
- The large open issue count shows heavy usage and active development, but it also means the product surface is broad and noisy.
- The open-core license split matters. Legal and procurement should review the `ee` licensing before serious company adoption.
- Session replay and user tracking can create privacy problems if consent, masking, and retention rules are sloppy.
- PostHog will not create product judgment. It gives evidence; the PO still has to ask better questions.
